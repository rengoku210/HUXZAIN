HUXZAIN MARKETPLACE DEVELOPMENT
PART 1 - UNIVERSAL LISTING SYSTEM, ORDER FLOW, CHAT SYSTEM & DELIVERY ARCHITECTURE
Before implementing individual categories, the entire platform should be built around a universal order and delivery architecture so that every category follows a consistent workflow while still allowing category-specific delivery methods.
Every listing on HUXZAIN should be created through a dynamic listing creation system.
When a seller clicks "Create Listing", the seller should first select the category and subcategory.
Examples:
Accounts
Gift Cards
Software
Game Currency
Digital Assets
Game Buddy
Coaching
Graphic Design
Programming
Website Development
Marketing
Hosting
Subscriptions
Business Services
Once the category is selected, the system should automatically display the relevant fields for that category.
Each listing should contain:
Listing Title
Cover Image
Gallery Images
Category
Subcategory
Description
Delivery Method
Delivery Time
Pricing
Revision Policy (where applicable)
Requirements From Buyer
Frequently Asked Questions
Additional Notes
The seller should also be able to choose whether the listing is:
Instant Delivery
Manual Delivery
Service-Based Delivery
Booking-Based Delivery
The delivery method selected by the seller will determine how the order proceeds after payment.

UNIVERSAL ORDER FLOW
All categories should follow the same order lifecycle.
Buyer places order.
Payment is completed.
Funds are held by HUXZAIN.
Seller receives notification.
Order room is created.
Buyer and seller are redirected into the order room.
Order room contains:
Order Information
Order Status
Internal Chat
File Uploads
Delivery Section
Dispute Button
Order Timeline
Review Section
Order Notes
The order room becomes the central location for the entire transaction.
No delivery should happen outside the order room.
No external communication should be required.
Everything should remain inside HUXZAIN.

ORDER STATUS SYSTEM
Every order should move through stages.
Payment Received
Seller Processing
Awaiting Buyer Requirements

Completed

Refund Rejected
Completed With Warning
The timeline should clearly display every action taken throughout the order.

INTERNAL CHAT SYSTEM
The chat system is one of the most important features of the marketplace.
Every order should automatically create a dedicated order chat room.
Only the buyer and seller involved in the order should have access.
Moderators and administrators should only gain access when a dispute is opened or manual review is required. Only superadmin/owner should have access to enter and interrupt chat . superadmin/owner should also have function to allot the chat to moderator or any employee .
The platform should not require staff to actively monitor chats.
Chats should simply be stored and made available if disputes occur.
Above every order chat display:
"For your protection, keep all communication, payments and deliveries inside HUXZAIN."
The chat system should support:
Text Messages
Images
Documents
Delivery Files
Screenshots
System Notifications
Order Updates
Messages should be timestamped.
Files uploaded through chat should be linked to the order.

Implement automated detection for:

The system should not require manual staff review unless a dispute is opened.

CHAT STORAGE POLICY
All order chats should be stored.
Recommended retention:
12 months minimum.
Chat messages are lightweight text data and generally consume very little storage compared to images and videos.
The major storage usage comes from file uploads.
Text conversations should remain accessible for at least one year.
File attachments may later receive separate retention policies depending on infrastructure costs.
All chats should remain searchable by administrators when disputes occur.

UNIVERSAL DELIVERY CENTER
Every order should contain a dedicated delivery panel.
Seller should never be forced to use chat for delivery.
Instead the platform should provide structured delivery submission.
Seller clicks:
Submit Delivery
The system then displays delivery options based on listing category.
Examples:

Any delivery submitted through this section should automatically generate a timeline entry.

This delivery architecture should serve as the foundation for every category implemented throughout the HUXZAIN marketplace.

ACCOUNT INSPECTION PERIOD SYSTEM
Inspection periods should not be selected by sellers.
Inspection periods should be automatically assigned by HUXZAIN based on category risk level and seller trust level.
The purpose is to protect buyers from:
* Account recoveries
* Fake deliveries
* Incorrect credentials
* Misleading descriptions
* Post-sale account issues
while still protecting legitimate sellers.

SELLER PLAN BENEFITS
Inspection periods may be reduced for trusted sellers.
This becomes one of the benefits of upgrading seller plans.

STANDARD SELLERS
Default inspection period applies.
No reduction.

PRO SELLERS
Inspection period reduced by approximately 25%.
Examples:
72 Hours → 54 Hours
48 Hours → 36 Hours

ELITE SELLERS
Inspection period reduced by approximately 40%.
Examples:
72 Hours → 43 Hours
48 Hours → 29 Hours

ENTERPRISE SELLERS
Inspection period reduced by approximately 50%.
Examples:
72 Hours → 36 Hours
48 Hours → 24 Hours

ADDITIONAL TRUST FACTORS
HUXZAIN administrators should have the ability to further reduce inspection periods for sellers who demonstrate:
* Long account history
* Large completed order count
* High ratings
* Low dispute rates
* Verified status
* Strong platform reputation

FUNDS RELEASE LOGIC
After delivery:
Order enters inspection stage.
Buyer receives notification.
Inspection countdown begins.
If buyer confirms completion:
Funds release immediately.
If buyer opens dispute:
Funds remain locked until resolution.
If inspection timer expires without dispute:
Order automatically completes.
Funds automatically release to seller.

ADMIN OVERRIDE
Super Admin and authorized moderators should have the ability to:
* Extend inspection periods
* Reduce inspection periods
* Hold funds
* Release funds
* Freeze suspicious transactions
This allows manual intervention for unusual cases.
The inspection system should be designed as a trust and risk management mechanism rather than a seller-selected setting.

HUXZAIN EARNINGS, RELEASE & WITHDRAWAL SYSTEM
The platform should not operate as a digital bank, savings account or long-term wallet.
The objective is to process marketplace transactions and release seller earnings, not store money indefinitely.

ORDER BASED EARNINGS SYSTEM
Every completed order should create its own earnings record.
Example:
Order #1001
Order Value: ₹1000
Platform Fee: ₹100
Seller Earnings: ₹900
Status: Pending Release
Each order should be individually tracked.

Each order should maintain its own earnings lifecycle.

COOLING PERIOD
After an order completes and inspection period ends, earnings should not become immediately withdrawable.
A cooling period should apply.
Purpose:
* Reduce fraud
* Handle delayed disputes
* Handle payment issues
* Reduce chargeback risk
Examples:
Standard Seller:
7 Days
Pro Seller:
5 Days
Elite Seller:
3 Days
Enterprise Seller:
1 Day
Super Admin should be able to modify these values.

WITHDRAWAL ELIGIBILITY
Only earnings that have completed:
Inspection Period
AND
Cooling Period
should become eligible for withdrawal.

WITHDRAWAL EXPIRY SYSTEM
HUXZAIN should not hold seller money indefinitely.
Once earnings become eligible for withdrawal:
Seller receives notification.
Examples:
Email
Dashboard Notification
Internal Notification
The seller is given a withdrawal claim window.
Recommended:
30 Days
If seller does not claim withdrawal within that period:
Withdrawal eligibility expires.

EXPIRED EARNINGS STATUS
When claim window expires:
Status becomes:
Withdrawal Expired
The earnings are not lost.
However they require manual reactivation.
Seller must submit request:
"Reactivate Withdrawal"
This prevents sellers from treating HUXZAIN as a long-term money storage service.

PLATFORM OBJECTIVE
Every important action on HUXZAIN should generate the correct email to the correct person at the correct time.
The email system should reduce confusion, increase trust, improve response times and ensure that buyers, sellers, staff and management remain informed throughout the entire lifecycle of every transaction.
HUXZAIN CORE EMAIL SYSTEM (PHASE 1)
Only important operational emails should be implemented initially.
The objective is to avoid overwhelming users while ensuring every important transaction, security event and dispute generates an email notification.
